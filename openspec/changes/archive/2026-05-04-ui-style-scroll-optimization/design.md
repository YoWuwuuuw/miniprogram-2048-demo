## Context

2048 微信小游戏的排行榜和"我的"页面存在两个问题：
1. **样式问题**：排行榜表头被标题遮挡，"我的"页面数据卡片偏小导致空旷
2. **滚动体验**：当前滚动是同步拖拽模式（`input.js` 的 `_onTouchMove` 直接发送累积 deltaY），没有惯性和弹性效果

当前滚动实现：`InputManager` 在 `_onTouchMove` 中计算 `startPos.y - pos.y` 作为累积偏移量，直接发送给场景。`_onTouchEnd` 时重置 `startPos`，滚动立即停止。

## Goals / Non-Goals

**Goals:**
- 排行榜表头与标题之间保持足够间距，不被遮挡
- "我的"页面数据展示更充实，减少空旷感
- 列表滚动有惯性：手指离开后继续滑动并逐渐减速
- 列表滚动有弹性：到达边界时有拉扯/回弹效果

**Non-Goals:**
- 不改变游戏核心逻辑
- 不引入外部滚动库
- 不改变排行榜数据结构或云端接口

## Decisions

### 1. 惯性/弹性滚动在场景层实现（不在 InputManager 层）

**选择**：保持 `InputManager` 只负责手势识别，在各场景的 `_onScroll` 和 `update(dt)` 中实现惯性和弹性逻辑。

**理由**：
- `InputManager` 是通用手势模块，不应耦合滚动物理效果
- 各场景可能需要不同的滚动参数（如最大滚动距离不同）
- 惯性需要 `update(dt)` 循环驱动，而 `InputManager` 没有 update 机制

**替代方案**：在 `InputManager` 中加入速度追踪和惯性驱动 → 耦合度太高，且需要改架构让 InputManager 有 update 循环。

### 2. 速度追踪方案

在场景中记录触摸移动的速度历史（最近几帧的 dy/dt），`_onTouchEnd` 时用最后几帧的平均速度作为初始惯性速度。

**实现要点**：
- `_onTouchMove` 时记录 `{dy, timestamp}` 到速度缓冲区（保留最近 3-5 帧）
- `_onTouchEnd` 时计算平均速度，启动惯性动画
- 惯性使用指数衰减：`velocity *= friction^dt`，friction 约 0.95-0.98

### 3. 弹性效果方案

使用"越界阻尼 + 回弹动画"：
- 滚动超出边界时，实际偏移量 = 边界值 + (超出量 * damping)，damping 约 0.3-0.5
- 手指离开后，如果在越界区域，使用弹簧动画回弹到边界值
- 惯性滑动到边界时，速度衰减更快（模拟摩擦）

**替代方案**：纯 CSS `overscroll-behavior` → Canvas 渲染不适用。

### 4. "我的"页面样式调整

- 用户数据卡片区域增大：增大字体、增加内边距
- 列表项高度从 52px 增加到 60px（卡片 56px + 4px 间距）
- 排行榜列表项保持 52px 不变（信息密度已经合适）

## Risks / Trade-offs

- **[性能]** update 循环中计算惯性可能影响低端设备 → 使用 requestAnimationFrame 自然节流，计算量极小
- **[手感调参]** 惯性摩擦系数和弹性阻尼需要实际测试调优 → 先用保守值，后续微调
- **[滚动冲突]** 惯性滚动中用户触摸应立即停止惯性 → 在 `_onTouchStart` 中重置惯性状态
