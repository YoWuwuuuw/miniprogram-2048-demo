## Context

2048 微信小游戏已完成功能开发，进入收尾阶段。当前代码结构清晰（game-core.js 纯逻辑层 + game-scene.js 场景层 + animation.js 动画层 + cloud.js 云端层），但测试中发现若干体验和稳定性问题需要统一修复。

## Goals / Non-Goals

**Goals:**
- 修复"我的"页面标题被遮挡的 UI 问题
- 修复方块移动动画在边界情况下跳过滑动直接合并的 bug
- 为排行榜数据添加缓存机制，减少不必要的云端请求
- 加强各模块的边界条件防护，提升稳定性

**Non-Goals:**
- 不改变游戏核心玩法逻辑
- 不重构现有架构
- 不新增功能特性

## Decisions

### 1. Profile 标题位置调整

**方案**：将 `titleY` 从 `screenHeight * 0.10` 调整为 `screenHeight * 0.12` 或更大值，确保标题在安全区域内。

**理由**：微信小程序顶部有状态栏和胶囊按钮区域，`0.10` 可能不够。通过微调比例值即可解决，无需引入复杂的安全区域计算。

### 2. 方块移动动画匹配逻辑修复

**问题根因**：`handleSwipe` 中通过"旧位置→新位置"的映射来创建移动动画。当多个相同值的方块存在时，`consumed` 标记的消费顺序可能出错——非合并方块先消费了本应属于合并的源方块，导致合并方块找不到正确的源位置，跳过动画直接定位。

**方案**：
- 强化合并方块的优先匹配：在第一轮处理合并时，不仅检查同列/同行，还要验证两个源方块在移动方向上确实是相邻的（中间无间隔）
- 对于无法匹配的方块，fallback 到同方向最近的未消费源方块，而非直接跳过动画
- 增加日志断言，在开发模式下检测匹配失败的情况

**替代方案**：重写整个匹配算法（基于图匹配）——过于复杂，收益不大。

### 3. 排行榜缓存策略

**方案**：在 `cloud.js` 中实现内存缓存层：
- 缓存结构：`{ data, timestamp }`
- TTL：60 秒（进入排行榜页面时，若缓存未过期则直接使用）
- 强制刷新：下拉刷新或手动触发时绕过缓存
- 缓存键：固定 key（因为排行榜数据是全局的）

**理由**：
- 内存缓存最简单，无需持久化
- 60 秒 TTL 平衡了实时性和请求频率
- 用户每次进入排行榜页面不会重复请求，但一分钟后会自动刷新

**替代方案**：
- localStorage 持久化缓存——排行榜数据变化快，持久化意义不大
- 无缓存但节流——无法保证短时间内多次进入时的数据一致性

### 4. 边界条件防护

**策略**：逐模块排查关键路径：
- `game-core.js`：`move()` 增加 board 参数校验（非空、4x4 结构）
- `storage.js`：`loadGameState()` 增加 try-catch，解析失败返回 null
- `cloud.js`：`fetchRankList()` 已有 try-catch，确保返回空数组而非抛异常
- `game-scene.js`：`_onMoveAnimationComplete` 防止重复调用（已有 `_moveAnimCalled` 标记，需确认生效）

## Risks / Trade-offs

- [缓存一致性] 60 秒内其他用户的分数变化不会实时反映 → 可接受，排行榜不要求毫秒级实时
- [动画匹配] 修改匹配逻辑可能引入新的边界情况 → 通过充分测试覆盖常见滑动模式
- [标题位置] 不同机型的安全区域高度不同 → 使用比例值而非固定像素，兼容性较好
